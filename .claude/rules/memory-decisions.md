# Decisions

Decisions made on this project, each with a date and a one-line why. Append; don't rewrite history.

- 2026-07-23 — **PLANNED recommendation-engine improvement: redesign the venue taxonomy into independent buckets.** Full spec: `docs/superpowers/specs/2026-07-23-venue-taxonomy-redesign.md`. Problem: `Pin.interests[]` is a dumping ground mixing venue-type / activity / vibe / occasion / descriptor, so the flat soft-score (`score.js`: activities matched on `interests`, restaurants on `cuisine`) conflates orthogonal signals (a café's `cozy` noise inflates intensity) and literally can't represent "romantic sushi date night" (restaurants have nowhere to hold a vibe). The 1,697 out-of-vocab tags from AI enrichment are the same mess quantified. Proposed 10 buckets — KEEP `category`(gate)/`cuisine`/`diet`(hard filter); SPLIT `interests[]` → `venueTypes[]`+`activities[]`+`vibes[]`+`occasions[]`; ADD `amenities[]`+`priceTier`(Int 1–4)+`neighborhood`(scalar). Each scores independently with tuned weights (cuisine 0.30, venueType 0.15, vibe 0.12, occasion 0.10, … — vibe/occasion low so they break ties, never override relevance). `normalizeTag()` + the `canonical→variants` vocab structure are KEPT (just register new bucket lookups). Objective buckets = scraper source-of-truth (Google/OSM); subjective buckets (vibe/occasion) = AI-only. Migration is additive→backfill(reclassify interests)→re-enrich→drop `interests`. NOT built — needs team review before the migration. This is the concrete plan for the still-open recommendation-quality gap the OSM enrichment surfaced.

- 2026-07-22 — **OSM-imported venues' `rating` + tags should be AI-generated, not left null/generic.** The `osmVenues.generated.js` import (~4,255 rows, parsed from an OpenStreetMap NorCal extract via `scripts/enrich/parse_osm_pbf.py`) landed venues with `rating: null` and coarse fallback interests (everything → `entertainment`/`casual`, food inferred from OSM category). These dominate the catalog (~4,280 rating-null vs ~399 curated after seeding) and dilute recommendation quality. **Direction:** enrich each OSM venue's `rating` and interest/cuisine/diet tags via AI calls (reuse the OpenRouter/Salesforce-gateway client in `services/ai/generation/`), replacing the null rating + generic tags with real signal. Not yet built — a batch enrichment job (cost-bearing: thousands of calls; needs batching, caching, idempotency, and a way to mark enriched rows). Related: the recommendation engine's `enrich/enrich.js` is still a no-op stretch seam; this is the concrete plan for it. Also relevant: today there's no `Pin.source`/`enrichedAt` column to distinguish OSM-vs-curated (currently inferred via `rating IS NULL`) — worth adding so enrichment is idempotent and re-runnable.
- 2026-07-22 — Trimmed **chain-flooding** from the seeded catalog: the OSM import added many branches of the same chain (50 Starbucks, 18 Subway, 17 Peet's, …). Deleted 288 redundant branches (kept 1 representative per chain — the rated/curated one where present, never one referenced by an itinerary stop), 71 chains collapsed, catalog 4679 → 4391 pins. NOT a true-duplicate issue (dedup by name+coords held: 0 exact dupes); these were distinct-location branches of chains. Caveat: done as a one-off DB delete, NOT reflected in `osmVenues.generated.js`, so a future `npm run seed:places` would re-insert them — the generated source or the seed loader needs a chain-cap to make it stick.

- 2026-07-22 — **Corrections to earlier entries (state as of the core audit).** (1) **Geocoding is now frontend-only, not backend.** The 2026-07-15/16 notes describing backend geocoding (Nominatim, then Geoapify) are superseded: `validateRecommendationInput` now *requires* each member's `startLocation` to already be `{latitude, longitude}`; the frontend resolves the address (Geoapify autocomplete) and sends coords. A member without coords ⇒ whole-request **400**, not a per-member silent drop or organizer hard-error. No backend geocoding code exists. (2) **The "diet no-op" and "openingHours proxy" limitations (2026-07-14 entry) are RESOLVED.** After the Pin/ItineraryStop schema split, `hoursOpen` is a real per-day JSON column (not a startTime/endTime proxy), and the catalog now carries 356 real `diets` entries so `passesDiet`/`ensureEveryDietCovered` actually apply (the unknown-diet-⇒-keep rule remains an intentional permissive choice, not a bug). `cuisine`/`diet`/`interests` are now explicit columns (`Pin.tags` was dropped), read via `mapVenue` as singular `cuisine`/`diet` in the engine's domain shape.
- 2026-07-22 — **Core-functionality audit + hardening pass** (5 parallel subagents: spec/backend/frontend/DB/AI). Fixed, in priority order: (a) **Critical** — Discover interest filter queried the dropped `Pin.tags` column → 500 on any `?interests=`; now matches `interests`/`cuisines`/`diets`. (b) Like endpoint gated to public-or-owner (was leaking private-draft metadata into a liker's dashboard). (c) Like AND bookmark toggles made race-safe via a shared `useLikeBookmark` hook (`frontend/src/hooks/`): per-id single-in-flight drain loop converging to the user's latest desired state — replaces the earlier per-click fire-and-forget that let spam-clicks leave the count/state wrong. (d) Wired 4 core user stories whose backends existed but frontends didn't: Edit/Delete/Copy on ItineraryPage (via `window.prompt/confirm`, no new modal files) + the "Created" dashboard carousel (was hardcoded `[]`); dashboard now reshapes `createdItineraries` so counts match Explore. (e) Discover cards' like/bookmark now hydrate + persist (were inert — props dropped in `ItinerariesGrid`). Robustness: `orderInItinerary` collision → 409 (was uncaught 500); `userController` username/email checks moved into `models/users.js` (was direct `supabase.from('User')`); AI client env/cert reads moved to `lib/aiClient.js` (was `process.env` in a service); `callAI`/`generateItinerary` given injectable seams + 15 new tests covering the previously-untested live-AI branch (good/hallucinated/malformed/infeasible) and prompt input caps; guarded `JSON.parse(localStorage)` via shared `lib/currentUser.js`; clamped session-expiry `setTimeout` against 32-bit overflow; HomePage spinner now clears for signed-out users. NOT done (deferred, needs shared-DB migration): FK indexes (`Like.itineraryId` etc.). 243 backend tests green, frontend lints + builds clean.
- 2026-07-16 — Enriched the Pin catalog with a **hand-curated static SF dataset** (~360 real places: 146 food + 213 activities/events), NOT an external API. Why: user tried/rejected OSM+Overpass and Google/Geoapify in turn; static data is deterministic, needs no key/network, reproducible. Files: `backend/prisma/data/sfPlaces/{restaurants,cafesBakeriesDesserts,parksGardensBeaches,cultureLandmarks,moreActivities}.js` re-exported flat via `index.js`; loaded by `backend/scripts/seedSfPlaces.js` (`npm run seed:places`). Grew in batches: ~250 → +100 activities/events (`moreActivities.js`) → +46 interactive nightlife/games (`nightlifeAndGames.js`: arcades, karaoke, dance clubs, sports/pool/skating courts, escape rooms, cocktail & live-music bars). Now **405 places (146 food / 259 activities)** — catalog deliberately skews toward things-to-do (a real day has more stops than meals). All entries SF-only (bbox lat 37.70–37.83, lon -122.52 to -122.35) — dropped a few Marin spots (Angel Island, Battery Spencer, Kirby Cove) that fell outside.
- 2026-07-16 — Made **`Pin.itineraryId` nullable** (migration `20260716210731_make_pin_itinerary_optional`) so catalog places exist as standalone pins with no parent itinerary, rather than under a fake "catalog" itinerary. Ripple: `pinsRepository.getAllPins()` filter widened from `itinerary: { isPublic: true }` to `OR: [{ itineraryId: null }, { itinerary: { isPublic: true } }]` so catalog pins are visible to the engine while private-draft pins stay excluded (Hawk Hill privacy test still green). `@@unique([itineraryId, orderInItinerary])` kept — Postgres treats NULLs as distinct so catalog pins (`orderInItinerary: 0`) don't collide.
- 2026-07-16 — Seed loader is **additive + idempotent**: dedupes new places against ALL existing pins by `name + coords(4dp)` before `createMany({ skipDuplicates: true })`, so re-runs insert 0 and it never touches `prisma/seed.js`'s demo data. Catalog pins get a neutral all-day window (08:00–22:00) + placeholder `locationImageUrl` (the only field without a real value). Why: user wants additive enrichment of the existing table, not a reseed.

- 2026-07-14 — Recommendation tunables live in `backend/config/recommendation.js`. Why: backend rules name WEIGHTS/FOOD_MIN/MAX/PRICE_LEVEL_USD as the trigger for a config module.
- 2026-07-14 — Exposed `AVG_STOP_DURATION_MIN` + `SHORTLIST_MULTIPLIER` instead of a flat `SHORTLIST_SIZE`. Why: Step 6 computes shortlist size from the time window, so a fixed constant would be a magic number.
- 2026-07-14 — Test runner is Node's built-in `node:test` (`npm test` = `node --test`). Why: zero dependencies, fits the free/open-source preference; no jest/vitest needed.
- 2026-07-14 — `passesDiet` requires a food place to serve EVERY diet present in the group (shared meal); unknown place-diet data ⇒ keep. Why: design says missing data must never silently drop a place.
- 2026-07-14 — No separate `Place` table for the recommendation engine's catalog; `Pin` (all pins, across all itineraries, deduped by name+coords) doubles as the seeded place data for Step 9. Why: team call — avoids a schema change/migration risking conflicts with the existing Itinerary/Pin model other work depends on; Pin already carries name/tags/price/coords which is enough to run the engine today.
- 2026-07-14 — `Pin`'s missing category/cuisine/diet/priceLevel/openingHours/rating are NOT added as new columns. `category` is derived from `tags` (a `food` tag ⇒ `restaurant`, else `activity`) as an interim stand-in for the still-open `INTEREST_MAP`/`CUISINE_MAP` canonical vocab (Step 0); cuisine/diet/rating/openingHours are left `undefined` and rely on the engine's existing missing-data handling. Why: keeps Pin schema-stable; the engine was already built from Step 2 onward to treat unknown place data as "keep, don't punish," so this is exactly the intended use case, not a workaround.
- 2026-07-14 — Backend confirmed as ESM (`"type": "module"` in `backend/package.json`, added by a teammate after Step 8 shipped). Converted all of `services/recommendation/**`, `config/recommendation.js`, and `prisma/seed.js` from CommonJS (`require`/`module.exports`) to `import`/`export` to match. Why: no choice — CJS syntax throws under `"type": "module"`; matches the rest of the codebase (`index.js`, `userController.js`) which was already ESM.
- 2026-07-14 — Added `backend/lib/prisma.js` as a shared `PrismaClient` singleton per `.claude/rules/backend.md`, used only by the new recommendation service (didn't touch `userController.js`'s inline client — out of scope, avoids merge risk with teammate code).
- 2026-07-14 — `POST /recommendations` is its own endpoint (not folded into `/ai-agent`) returning `{ shortlist, constraints }` directly. Why: AI sequencing (`/ai-agent`) isn't built yet; Step 9's done-when criterion only requires the endpoint to return the shortlist, and this keeps the recommendation engine independently testable over HTTP before AI wiring exists.
- 2026-07-14 — Revisited the "no new Pin columns" call: added `rating` (nullable `Float?`) as the one genuinely new column, since nothing else can derive it. `category`/`cuisine`/`diet` still aren't columns — derived from `Pin.tags` via a shared vocab (`config/tagVocab.js`), and `openingHours` is approximated from the Pin's own scheduled `startTime`/`endTime` (converted through `America/Los_Angeles`, since `TIMESTAMP(3)` columns store bare UTC instants and `.getHours()` would be server-timezone dependent). Why: keeps the schema change minimal (one column) while making cuisine/diet/category actually usable from data that already exists, at the cost of `openingHours` now being a proxy (see Known Limitations in the roadmap for the tradeoffs).
- 2026-07-14 — Fixed the privacy leak found during Step 9 QA: `getAllPlaces()` now scopes its `prisma.pin.findMany()` to `where: { itinerary: { isPublic: true } }` instead of querying every Pin. Why: confirmed live that a Pin on someone's private draft itinerary ("Hawk Hill") surfaced in a stranger's recommendations — a real privacy bug, not a hypothetical; added a regression test in `services/recommendation/index.test.js` to lock it in, and had to swap the DB-integration test fixture's `Sam` interest from `hiking` (only matched by the now-excluded private Pin) to `nature` (matched by a public one) since the old fixture was inadvertently asserting the buggy behavior.
- 2026-07-14 — Extracted `middleware/auth.js` (`requireAuth`) + `lib/supabase.js` as the shared auth check, instead of inlining another copy of `getAuthUser`/`authenticateProfile` into `recommendationController.js`. Why: `.claude/rules/backend.md` already names `middleware/auth.js` in the target structure and lists auth as shared-middleware functionality; `POST /recommendations` is the first consumer. Deliberately did NOT retrofit `itineraryController.js`/`pinController.js`/`userController.js`'s existing inline copies onto this shared version — out of scope for an auth-bug fix, avoids merge risk on teammate-owned files.
- 2026-07-14 — `postRecommendations` wraps `getRecommendations()` in try/catch, logs, and returns `500 { error: 'Failed to generate recommendations' }` on failure, rather than adding a global Express error-handling middleware. Why: matches the existing per-controller try/catch convention already used for Prisma errors in `userController.js` (registerUser/updateUser); no global error handler exists elsewhere in the codebase to hook into instead.
- 2026-07-15 — Models are now REQUIRED for all DB access (updated `.claude/rules/backend.md` from "optional"). Created `backend/models/{users,itineraries,pins,likes,bookmarks}.js` (one thin wrapper per table, all importing the `lib/prisma.js` singleton) and refactored `user`/`itinerary`/`pinController.js` so no controller calls `prisma` directly. Why: user directive — one model per Prisma table, controllers do HTTP concerns only. Repository carve-out documented: `services/recommendation/placesRepository` stays service-owned (maps rows to a domain shape, not a thin table wrapper).
- 2026-07-15 — Reversed the 2026-07-14 "don't retrofit inline auth" decision: `user`/`itinerary`/`pinController.js` no longer inline `getAuthUser`/`authenticateProfile`/`new PrismaClient()`/`createClient` — they now run behind `middleware/auth.js`'s `requireAuth` (applied in the routes) and read `req.user`. `middleware/auth.js` itself now resolves the profile via `users.findByAuthUserId` instead of a direct `prisma` call. Why: user directive to reduce duplication; merge-risk concern that justified leaving the copies is outweighed now that the consolidation is the explicit ask. 97 tests green, boots clean.
- 2026-07-15 — Recommendation engine gains a "Stage 0" (design + roadmap docs only so far, not built): compute a fair group **meeting point** as the **geometric median** (Weiszfeld) of members' start locations, then hard-filter places to those within `trip.travelRadius` **miles** (straight-line/haversine) of it. Why (meeting-point method): geometric median minimizes TOTAL group travel and resists one far-away member dragging the anchor — plain centroid (minimizes squared distance) doesn't, which is the exact unfairness the client flagged. Chose miles per client's form. `startLocation` is free text ⇒ needs geocoding (free OSM **Nominatim**, cached, ≤1 req/sec) to `{latitude,longitude}` before the engine runs; un-geocodable address = hard error to organizer, not silent drop. Geo math (`haversineMiles`/`centroid`/`geometricMedian`) goes in `backend/utils/geo.js` (reused by AI sequencing for travel distances); `withinRadius` wrapper in `helpers.js`. `meetingPoint`+`travelRadius` also added to `recommend()`'s returned `constraints` so the AI anchors the day's first stop there. Docs updated: recommendation-engine design (Stage 0) + roadmap (new Step 2.5), ai-design, architecture flow, ai-sequencing roadmap.
